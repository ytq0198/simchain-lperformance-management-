## 1 实验目的

1. 理解联盟链在**多机构协作**场景下的部署方式与角色划分。  
2. 掌握 Hyperledger Fabric 中**通道、Peer、Orderer、MSP、链码**的基本概念与联调流程。  
3. 设计并实现「学生成绩」的**链上存证、查询、更正溯源与验真**等业务流程。  
4. 验证区块链相对传统中心化存储在**不可篡改、可追溯**方面的特性，并能结合业务说明其代价与边界。

### 2.12 资源与常见问题（4 GB 内存虚拟机）

| 现象 | 建议 |
|------|------|
| 容器启动慢或 OOM | 关闭虚拟机内不用的浏览器标签、IDE；减少同时运行的 Peer 数；使用 LevelDB 而非 CouchDB（若课程允许） |
| 磁盘不足 | `docker system prune`（谨慎，会删未使用镜像）；清理 `apt` 缓存 |
| 端口被占用 | `sudo ss -tlnp` 查找占用进程，或修改 `docker-compose` 端口映射 |
| 无 sudo 权限使用 docker | 用户须加入 **`docker`** 组并**刷新登录会话**；详见下文 **§2.12.1** |

#### 2.12.1 Docker 组与 `docker.sock`（`permission denied` 完整处理）

**现象**：**`docker ps`**、**`./network.sh up`** 等报 **`permission denied … /var/run/docker.sock`**。

**步骤**：

1. **`sudo usermod -aG docker <你的用户名>`**（例：`tony`）。  
2. **`getent group docker`** 确认用户名出现在组列表末尾。  
3. **刷新组**：**注销 Ubuntu 再登录**（图形界面电源菜单选 **Log Out**，勿选 **Suspend**）；或当前终端执行 **`newgrp docker`**。  
4. **不加 sudo** 执行 **`docker ps`** 验证。  
5. 若仍失败，检查 **`ls -l /var/run/docker.sock`**（常见为 **`root docker`**、**`srw-rw----`**），必要时 **`sudo chown root:docker /var/run/docker.sock`**、**`sudo chmod 660 /var/run/docker.sock`**，再 **`sudo systemctl restart docker`**。

**说明**：**`newgrp docker`** 仅影响**当前终端**；桌面**新开的终端**有时仍带旧组信息，需**再注销一次**或在新终端里再 **`newgrp docker`**。**`network.sh`** 请在已能无 sudo 执行 **`docker ps`** 的同一环境中运行。

**可选截图位**：**`groups`** 含 **`docker`**、**`docker ps`** 成功、**`./network.sh up`** 无 socket 报错。

**可选：增加交换分区（swap）缓解内存峰值（按需执行）**

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
# 若要开机自动挂载：将 /swapfile none swap sw 0 0 写入 /etc/fstab（需理解风险后再做）
free -h
```

### 2.14 步骤释义：`./network.sh deployCC` 在做什么？

**一句话：** 把指定目录里的**智能合约（链码）**按 Fabric **2.x 生命周期**装到通道 **`mychannel`** 上，让两个组织的 Peer 都能**背书/执行**同一份链码，并在需要时执行**初始化交易**。

**脚本大致完成的工程步骤（与 Fabric 概念对应）：**

1. **整理依赖**：在链码目录执行 **`go mod vendor`**（脚本内调用；**须使用与链码容器一致的 Go 版本**，本实验为 **Go 1.18**，避免 `inconsistent vendoring`）。  
2. **打包（package）**：`peer lifecycle chaincode package …` 生成 **`*.tar.gz`**（如 **`basic.tar.gz` / `score.tar.gz`**，标签形如 **`链码名_版本`**）。  
3. **安装（install）**：在各 Peer 上 **`peer lifecycle chaincode install`**，由 Peer 在 Docker 内**编译/构建链码镜像**并登记 **package id**。  
4. **批准定义（approve）**：各组织分别 **`approveformyorg`**，同意在通道上使用同名的链码定义（版本、背书策略、是否需 init 等）。  
5. **提交定义（commit）**：满足策略后 **`commit`**，通道上该链码定义**生效**。  
6. **初始化（可选）**：本实验使用了 **`-cci InitLedger`**，部署完成后会发起带 **`--isInit`** 的调用，执行链码里的 **`InitLedger`**，写入**示例资产数据**（教学用，与成绩业务无关，但流程与以后部署成绩链码相同）。

**与成绩大作业的关系：** `basic` 为 **fabric-samples** 自带的流程演练；**`score`** 为本项目仓库 **`score-chaincode/`**（`PutScore` / `GetScore`）在 **`mychannel`** 上的**实际部署实例**；后续可在同通道上升版或增加新链码名。


**说明**：**`status":"REVOKED"`** 表示逻辑作废；**`VerifyScore`** 对作废记录返回 **`match":false`**、**`reason":"record is revoked"`**，符合「作废后不可再按 ACTIVE 验真为真」的设计。

**重复作废（预期报错）**：若再次对同一键执行 **`RevokeScore`**，将出现 **背书失败 / status:500**，提示 **`score already revoked for key SCORE_2021003_PHYS101_2024-3`**，属链码**防重复作废**的正常行为，**无需修复**。

## 9 遇到的问题与解决方案

| 问题现象 | 原因分析 | 解决办法 |
|----------|----------|----------|
| `E: Package 'docker-ce' has no installation candidate` | 未正确添加 Docker CE 源或网络受限 | 改用 **`sudo apt install docker.io`**；或修复官方源后重试 |
| `Get "https://registry-1.docker.io/...": ... Timeout` | 访问 Docker Hub 受限 | 配置 **`/etc/docker/daemon.json`** 中 **`registry-mirrors`**；并优先 **`docker load` 离线 `fabric-*.tar`** |
| `docker-compose: command not found` | 未安装 v1 的 `docker-compose` 可执行文件 | **`sudo apt install docker-compose`** |
| `unknown shorthand flag: 'f' in -f` | `network.sh` 把 **compose** 当成了 **`docker`**（如错误包装脚本或 **`CONTAINER_CLI_COMPOSE=docker`**） | 删除错误的 `/usr/local/bin/docker-compose`；安装 apt 版 **`docker-compose`**；**`unset CONTAINER_CLI_COMPOSE`** |
| `LOCAL_VERSION` 与 `DOCKER_IMAGE_VERSION` 不一致 | **`fabric-tools:latest`** 被拉成与本地 `peer` 不同的大版本 | 删除错误 **`latest`** 标签后，将 **2.4.6** 对应镜像 **`docker tag …:latest`**，与其它 **`hyperledger/fabric-*:latest`** 一并指向 **2.4.6** 线 |
| `docker ps` / `./network.sh` 报 **`permission denied … docker.sock`** | 用户不在 **`docker`** 组，或已加组但**会话未刷新** | **`sudo usermod -aG docker <用户名>`** → **`getent group docker`** 确认 → **注销再登录**或 **`newgrp docker`** → **`docker ps`** 验证；仍失败则检查 **`/var/run/docker.sock`** 属组是否为 **`docker`**（见 **§2.12.1**、`score-chaincode/README-VM.md` 中「Docker：permission denied」） |
| `go mod vendor` 报 **`mkdir .../vendor: permission denied`**；`ls -ld ~/work/score-chaincode` 为 **`dr-xr-xr-x`** | 从 **Windows 拖拽/拷贝**进 VMware 时，目录常被设成**属主无写权限** | **`chmod -R u+w ~/work/score-chaincode`**（或 **`~/work`**）；或配置 **`~/.bashrc`** 中的 **`fixwork`** 函数，拖完执行一次（详见 **`score-chaincode/README-VM.md`** 开头「宿主机拖拽」一节） |
| `go mod vendor` / Docker 构建报 **`inconsistent vendoring`** / **`not marked as explicit in vendor/modules.txt`** | **`go.mod` 中 `go 1.14`** 与 **Go 1.18** 及 **ccenv** 的 **`-mod=vendor`** 校验不一致 | **`go.mod` 改为 `go 1.18`**；删除 **`vendor/`** 后 **`go mod tidy`** → **`go mod vendor`**（见 **`README-VM.md`**） |
| `go mod vendor` 提示 **updates to go.mod needed** 或 **`score.tar.gz` 未生成** | 未先执行 **`go mod tidy`** 或 **PATH 未指向 Go 1.18** | 在 **`~/work/score-chaincode`**：`export PATH="/usr/local/go118/bin:$PATH"` → **`go mod tidy`** → **`go mod vendor`** → 再 **`deployCC`** |
| `invoke` 显示成功但 `query` 仍 **not found** | **`peer chaincode`** 使用 **`{"function":"PutScore","Args":[…]}`** 时，与 **contract-api** 在 CLI 下的入参习惯不一致，可能未产生预期读写 | **写入与查询统一使用**：`-c '{"Args":["PutScore","学号","课程","学期","分数"]}'`、`-c '{"Args":["GetScore","学号","课程","学期"]}'`（见附录 A） |
| `npm run build`（`backend`）报 **`Client` is not assignable to `GrpcClient`** | **`@grpc/grpc-js` 1.6.x** 与 **`@hyperledger/fabric-gateway` 1.11** 所依赖的 **gRPC** 类型不一致 | 在 **`package.json`** 将 **`@grpc/grpc-js`** 升为 **`^1.14.0`**；在 VM 删除 **`node_modules`** 后 **`npm install`**（勿拖 Windows 的 **`node_modules`**） |
| **`npm start`** 报 **`EADDRINUSE :::3000`** | 上次 **`npm start` / nohup** 未退出，端口仍被占用 | **`fuser -k 3000/tcp`** 或 **`kill`** 对应 **PID** 后再启动；前台运行时 **`Ctrl+C`** 正常退出 |
| **`nohup … > ~/work/backend/server.log`** 报 **Permission denied** | 从 Windows 拖入的 **`~/work/backend`** 对当前用户**不可写** | **`sudo chown -R $USER:$USER ~/work/backend`**；或日志改 **`~/score-backend.log`** / **`/tmp/…`** |
| **`/api/health`** 正常但 **`fabricConfigured":false`** | **`.env` 未加载**（如在 **`~`** 启动旧进程）或 **`FABRIC_CRYPTO_BASE` 为空** | 使用仓库中 **`server.ts`** 的 **`dotenv` 显式路径**；确认 **`~/work/backend/.env`** 中已填 **Linux 绝对路径**；重启进程 |
| 例：链码背书失败 | 背书策略与提交组织不匹配 | （填写） |
| 例：容器启动失败 | 端口占用/内存不足 | （填写） |
