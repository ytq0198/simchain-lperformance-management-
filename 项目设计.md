# 学生成绩信息存证与查询 — 项目设计说明

> 选题：**围绕学生成绩信息存证与查询实现一个成绩管理应用**  
> 技术栈约束：**Hyperledger Fabric** 联盟链 + 链码（智能合约）+ 应用层（SDK）  
> 运行环境：**VMware 虚拟机 / Linux**（Ubuntu 20.04 / 22.04 均可；你当前为 **20.04 desktop**）

本文档在 `deepseek设计思路.txt` 与 `gemini设计思路.txt` 的基础上做取舍：保留**可答辩、可演示、可写进报告**的区块链亮点，避免在 16 学时大作业中过度堆叠运维组件导致无法在虚拟机内按期完成。

---

## 一、设计目标与对照课程要求

| 课程要求关键词 | 本设计中的落点 |
|----------------|----------------|
| 上链 | 成绩写入/更正/作废均以交易形式提交，由 Orderer 排序、Peer 落账 |
| 可追溯 | 使用 Fabric 对同一键的**版本历史**（如 `GetHistoryForKey` 或应用层按 TxID 聚合）展示「录入 → 更正」链路 |
| 不可篡改 | 不在链上物理删除记录；错误成绩采用**作废 + 新记录关联**或**状态字段标记** |
| 多方协作 | 至少两个 MSP（组织）：如 **教务处（权威）** 与 **学院/教师侧（录入）**；可选第三角色 **第三方验真**（单独 MSP 或同链不同证书属性） |
| 存证与查询 | 链上保存成绩资产与关键元数据；**按学生/课程/学期**查询与**验真**接口 |

## 二、总体架构（推荐「双组织 + 单通道」为基线）

### 2.1 为何不全盘采用「四组织 + 双通道 + 五个 Orderer」

DeepSeek 方案工程完整度高，但在**单台虚拟机**上容易出现：内存不足、证书与通道配置错误率高、联调周期长。建议：

- **基线版（必做，易验收）**：2 组织、1 通道、1 Orderer（Raft 单节点或 Solo 教学场景）、每组织 1～2 个 Peer、CouchDB 或 LevelDB 二选一。  
- **加分版（选做）**：增加第三个组织（用人单位/第三方）、或增加**私有数据集合（Private Data Collection）**、或增加**简单复核状态机**。

### 2.2 逻辑架构图（文字版）

```
┌─────────────┐     HTTPS/JSON      ┌──────────────────┐
│  Web 前端   │ ◄──────────────────► │  应用后端 (API)   │
│ Vue/React   │                     │ Node.js / Go     │
└─────────────┘                     │  Fabric Gateway   │
                                    └────────┬─────────┘
                                             │ gRPC
                    ┌────────────────────────┼────────────────────────┐
                    ▼                        ▼                        ▼
              ┌──────────┐            ┌──────────┐              ┌──────────┐
              │ Peer Org1│            │ Peer Org2│             │ Orderer  │
              │ (教务处)  │            │ (学院)   │             │          │
              └──────────┘            └──────────┘              └──────────┘
                    │                        │
                    └────────── 同一通道 (如 ScoreChannel) ──────────┘
```

**Gemini 思路采纳点**：双组织联盟链、复合主键/资产模型、**撤销/冲正式更正**体现「不可删除但可治理」。  
**DeepSeek 思路采纳点**：**批量录入**、**审计字段/审计事件**（可先简化为链码内写扩展状态或单独 Audit key）、**一次性授权验真**（应用层实现即可，链上存哈希更专业）。

## 三、链上数据模型与智能合约设计

### 3.1 资产（建议键设计）

为便于查询与历史溯源，主键建议采用可读复合键（链码内拼接）：

- **主键示例**：`SCORE | {studentId} | {courseId} | {semester}`  
- **值（JSON）建议字段**：

| 字段 | 说明 |
|------|------|
| `studentId` | 学号（链上可明文，若学校有合规要求可改为仅哈希上链） |
| `courseId` / `courseName` | 课程标识与名称 |
| `semester` | 学期 |
| `score` | 分数（若做私有数据集合，则公开账本仅存 `scoreHash`） |
| `status` | `ACTIVE` / `REVOKED` / `SUPERSEDED` 等 |
| `prevScoreKey` | 若本条为更正产生，指向被替代记录 |
| `teacherId` / `operator` | 操作者标识（来自客户端身份或参数+链码校验） |
| `updatedAt` | 时间戳 |
| `remark` | 可选备注 |

### 3.2 核心链码接口（Go，基线版）

| 方法 | 作用 | 权限思路 |
|------|------|------------|
| `InitLedger` | 可选：写入演示数据 | 管理员/初始化脚本 |
| `CreateScore` | 首次录入 | 教务处或学院角色（见下节） |
| `UpdateScore` | 更正：旧记录标记 `SUPERSEDED` 或 `REVOKED`，新记录关联 | 教务处背书或 Org1+Org2 背书策略 |
| `RevokeScore` | 作废错误记录（不删除 key 的历史） | 教务处 |
| `QueryScore` | 单条查询 | 学生本人 / 教师本学院 / 教务处（用证书属性或后端过滤） |
| `QueryScoresByStudent` | 按学号列表 | 同上 |
| `GetScoreHistory` | 返回该键的版本历史摘要 | 授权角色 |
| `VerifyScore` | 输入 `studentId,courseId,semester,claimedScore` 返回是否一致 | 可设计为公开查询或需应用层 token |

**说明**：课程大作业中「学生只查本人」若全在链码用 ABAC 做强校验，开发与演示成本较高。推荐折中：

- **链码**：校验「是否教务/学院角色」对写操作强约束；读操作按组织做粗粒度限制。  
- **应用后端**：结合业务登录态，只把学生本人学号发往链上查询（**链下身份 + 链上存证**是常见工程组合）。

若追求加分：再引入 **Fabric Identity 的 OU/attribute**（Gemini 提到的 `cid`）对 `CreateScore` 做 ABAC。

### 3.3 背书策略（示例）

- **成绩写入/更正**：`AND('OrgEduAdmin.peer', 'OrgCollege.peer')` 或 `OR` 组合按老师要求调整。  
- **查询**：默认通道读请求由背书节点执行即可；若使用私有数据，需配置 collection 并注意 endorser 选择。

### 3.4 私有数据集合（选做加分）

- 账本公开：`scoreHash = SHA256(学号+课程+学期+分数+盐)`  
- 私有集合：存明文分数或敏感字段  
- 第三方验真：由授权方提供明文，链码比对哈希返回 `true/false`（**不直接暴露他人成绩**）

## 四、应用层设计

### 4.1 后端（推荐 Node.js + Fabric Gateway）

- **连接管理**：进程内单例 Gateway（避免每请求新建连接，Gemini 建议合理）。  
- **身份**：为 Org1、Org2 各准备钱包用户（WALLET），演示时在 `.env` 切换调用身份。  
- **REST API（示例）**  
  - `POST /api/scores` 录入  
  - `PUT /api/scores/:key` 更正  
  - `GET /api/scores/student/:id` 查询  
  - `GET /api/scores/:key/history` 溯源  
  - `POST /api/verify` 验真（可选：带一次性 `token`）  
  - `GET /api/chain/info` 区块高度、最新交易摘要（配合前端「简易区块看板」）

### 4.2 前端（Vue3 或 React 均可）

**DeepSeek** 的「控制台分角色」与 **Gemini** 的「仪表盘 + 时间轴」建议合并为：

- 教务处：录入、更正、作废、批量导入（CSV 解析在**后端**完成，链码循环写入或批量提案）。  
- 学院教师：录入、查询本院/本班（可由后端参数过滤）。  
- 学生：仅查询本人、查看某门课**历史版本时间轴**。  
- 第三方验真页：输入学号+课程+学期+声称分数 → 显示「一致/不一致」。

**本仓库落地（2026-05-14 起）**：宿主机目录 **`frontend/`** 已按 **§十二** 选型实现 **Vue 3 + Vite + TypeScript + Vue Router + Element Plus + Axios**；通过环境变量 **`VITE_API_BASE`**（开发默认见 **`frontend/.env.development`**）指向虚拟机上的 **`backend/`**（默认端口 **3000**）。顶栏 **演示角色**（教务处 / 教师 / 学生 / 第三方）仅控制菜单与写路由拦截，**不**替代 Fabric 侧多钱包切换。未实现项：**`/scores/import` 批量导入**、**`/student` 独立页**、**`/api/chain/summary` 类区块高度接口**（Dashboard 当前仅消费 **`GET /api/health`** 已有字段）。

### 4.3 事件监听（选做）

- 使用 Fabric 区块/合约事件，在成绩确认后推送 WebSocket 通知前端「上链成功」。报告里可截图证明「异步确认」特性。

## 八、风险与虚拟机资源配置

- **内存**：Fabric + CouchDB + 多 Peer 建议 **≥ 4GB**；紧张时减 Peer 数量、用 LevelDB。  
- **与你当前虚拟机对齐**：截图配置为 **内存 4GB、CPU 8 核、磁盘 40GB、NAT**，可跑通教学用网络，但**不要**同时开 Grafana/Prometheus/过多 Peer。建议：**每组织 1 个 Peer**、**Orderer 1 个**、世界状态优先 **LevelDB**；CouchDB 仅在课程强制富查询时再开。  
- **磁盘**：Docker 镜像与日志增长快，预留 **≥ 20GB** 空闲。  
- **时间**：配置文件错误是最大风险；先跑通 `test-network` 再改组织名与通道名，比从零手写 network 更稳。

## 十一、Fabric 要装在哪？必须装在虚拟机里吗？

### 11.1 结论（按大作业验收场景）

| 内容 | 建议安装 / 运行位置 | 原因 |
|------|---------------------|------|
| Docker、Fabric 网络（Peer/Orderer/CA）、链码容器、老师提供的 **Fabric 二进制与镜像**、启动网络的脚本 | **必须在你用于演示的 Ubuntu 虚拟机里** | 作业要求在 Linux VM 上实现；链路与路径按 Linux 设计；答辩时评委通常认「VM 里一键起网 + 浏览器演示」。 |
| 代码编辑（Cursor / VS Code） | **可在 Windows 宿主机** | 仅编辑文件；通过 **共享文件夹、Git、SCP** 与虚拟机同步即可。 |
| 在 Windows 上单独装一套 Fabric 跑完整网络 | **一般不推荐** | WSL2 可跑但环境与课程/老师脚本不一致时排错成本高；除非老师明确要求 Windows 部署。 |

### 11.2 老师发的安装包怎么放

在虚拟机内任选**固定目录**即可，例如：

- `~/fabric/` 下解压：`bin`、`config` 与脚本；在 `~/.bashrc` 里 `export PATH=$PATH:$HOME/fabric/bin`。  
- 或按课程实验手册指定路径（**以老师文档为准**）。

关键是：**PATH 能找到 `peer`、`configtxgen`，Docker 能拉到所需镜像**；与「装在 C 盘还是 D 盘」无关，在 Linux 里就是「装在哪个用户目录」。

### 11.3 小组作业一个人做

在报告「分工说明」里如实写：**本人承担网络、链码、后端、前端与测试**即可；若需交分工表，可写「组员 A：本人全流程；其余组员未参与」以符合诚信要求（具体措辞按学校规定调整）。

## 十二、前端界面设计（成绩管理应用）

面向 **4GB 内存虚拟机**：前端只做**静态构建 + 由后端提供 API**，开发时可在 Windows 上 `npm run dev`，**联调与演示**建议在 VM 内或 VM 访问宿主机 IP（NAT 下注意端口转发），避免「仅宿主机可访问 localhost」导致答辩时打不开。

### 12.1 技术选型（二选一即可）

| 方案 | 栈 | 说明 |
|------|-----|------|
| A（推荐上手快） | **Vue 3 + Vite + TypeScript + Element Plus** | 表格/表单多，教务类界面成熟。 |
| B | **React 18 + Vite + TypeScript + Ant Design** | 与方案 A 等价，按个人熟悉度选。 |

HTTP 客户端：**Axios**；路由：**Vue Router / React Router**；状态：页面少可用 **组合式 API / useState**，不必强上 Pinia/Redux。

### 12.2 信息架构与路由

| 路由路径 | 页面名称 | 主要用户 | 功能要点 |
|----------|----------|----------|------------|
| `/` 或 `/login` | 演示入口 | 全员 | **演示用角色切换**（下拉：教务处 / 教师 / 学生 / 第三方），不实现真实校园统一身份亦可；后端按角色选用不同 Fabric 钱包调用。 |
| `/dashboard` | 链网概览 | 全员 | 展示后端返回的 **通道名、最新区块高度、最近 N 条交易 ID 摘要**（加分项，数据来自自定义 REST）。 |
| `/scores` | 成绩列表 | 教务、教师 | 表格：学号、课程、学期、分数、状态、更新时间；支持按学号/学期筛选。 |
| `/scores/new` | 录入成绩 | 教务、教师 | 表单：学号、课程代码、课程名、学期、分数、备注；提交 → `POST /api/scores`。 |
| `/scores/:key` | 成绩详情 | 教务、教师、学生 | 展示单条 JSON；按钮「查看链上历史」。 |
| `/scores/:key/history` | 溯源时间轴 | 教务、学生 | 纵向时间轴：版本号、分数、交易 ID、时间戳、操作说明（对应 `GetHistoryForKey` 或后端封装）。 |
| `/scores/import` | 批量导入 | 教务、教师 | 上传 CSV，由**后端解析**后循环调链码；前端只显示进度与结果列表。 |
| `/verify` | 第三方验真 | 第三方 | 表单：学号、课程、学期、**声称分数** → `POST /api/verify` → 显示「与链上记录一致 / 不一致」及简要说明（不写他人完整成绩单）。 |
| `/student` | 我的成绩 | 学生 | 固定演示学号或登录后学号，只读列表 + 进入历史页。 |

### 12.3 页面布局（统一框架）

- **顶栏**：应用标题「成绩链上存证系统」+ 当前角色标签 + 切换角色。  
- **侧栏**：按角色 **过滤菜单**（学生不显示「录入」「批量导入」）。  
- **内容区**：白底卡片（Card）承载表格或表单，表格支持加载中与错误提示（Message）。

### 12.4 关键交互与校验

- 提交前前端校验：分数范围（如 0–100）、学号非空、学期格式统一（如 `2024-1`）。  
- 所有写操作展示 **交易 ID** 或后端返回的 `txId`（便于写进实验报告与截图）。  
- 错误展示：网络错误 / 背书失败时显示后端返回的 `message`，避免只弹「Error」。

### 12.5 视觉与报告截图位（无需复杂美工）

- 主色：**深蓝 `#1a5fb4`** + 中性灰背景 `#f5f7fa`；成功/失败用绿/红文字即可。  
- 报告建议截图：**登录与角色切换**、**成绩列表**、**录入成功含 txId**、**历史时间轴至少两版本**、**验真页两种结果**、**Dashboard 区块高度**。

### 12.6 与后端的约定（便于前后端并行）

前端只认 **REST JSON**；**路径、查询参数与请求体字段以仓库 `backend/README.md` 为唯一契约**（下列为早期规划示例，若与 README 不一致，**以 README 为准**）。

- `GET /api/health`（已实现；响应含 **`channel` / `chaincode` / `fabricConfigured`** 等）  
- `GET /api/scores?studentId=&courseId=&semester=` → 链码 **`GetScore`**（单条当前态）  
- `GET /api/scores/history?studentId=&courseId=&semester=` → **`GetScoreHistory`**  
- `POST /api/verify`，body：`{ studentId, courseId, semester, claimedScore }`  
- `POST /api/scores`，body：`{ studentId, courseId, semester, score }`（**无**独立 `courseName` 字段；与当前链码一致）  
- `POST /api/scores/correct`，body：`{ studentId, courseId, semester, score, remark }`  
- `POST /api/scores/revoke`，body：`{ studentId, courseId, semester, remark }`  
- **未在 `backend/` 提供**：`GET /api/chain/summary`、`PUT /api/scores/:key` 等；若报告或界面需要「区块高度 / 最近 tx 列表」，需在 **`backend/`** 增路由后再接前端。

## 十三、企业化增强（结合 DeepSeek / Gemini，按你当前虚拟机可落地）

> **现状**：Ubuntu 20.04、约 **4GB RAM**、**test-network**（Org1/Org2 + 单 Orderer + LevelDB）、**Fabric 2.4.6**、链码 **`basic` + `score`（PutScore/GetScore 已通）**。  
> **原则**：企业化主要体现在 **治理模型（状态机）、可审计、API 契约、身份与职责分离、可复现部署与验收**；**不**在单台作业机上复制 DeepSeek 的「四组织 + 双通道 + 五 Orderer + Prometheus/Grafana + ELK」。

### 13.1 从两份思路里「该拿」与「该扔」

| 来源 | 建议保留（写入报告可讲故事） | 建议暂不实施（与 4GB / 学时冲突） |
|------|------------------------------|-----------------------------------|
| **DeepSeek** | 批量录入（**后端解析 CSV**，链码循环或分批提案）、**审计字段/操作者/时间戳**、验真 API、**分角色控制台**、一键脚本思想（收敛为 `scripts/*.sh` + README） | 四组织、双通道、五 Raft Orderer、每组织双 Peer、独立审计通道、Kong、RabbitMQ、Prometheus+Grafana 常驻 |
| **Gemini** | **冲正/作废（逻辑删除）**、**`GetHistoryForKey` 溯源**、简易 **Dashboard（区块高度 + 最近 txId）**、可选 **哈希验真**（链上存 `sha256`，验真时比对） | 国密 BCCSP（除非课程硬性要求；否则调研成本高） |
| **二者** | **私有数据集合（PDC）**：仅当报告需要「隐私加分」且你愿意改 lifecycle 部署时再上；否则用 **哈希 + 应用层授权** 已够答辩 | 在链码里做复杂统计/异常检测（占内存、难测） |

### 13.2 「企业化」最小闭环（推荐你作为项目目标）

1. **链上资产生命周期**：`ACTIVE` →（更正）`SUPERSEDED` 或 `REVOKED` + 新键/新记录关联 `prevScoreKey`（与 §3.1 表一致）。  
2. **链上审计最小实现**：每条 `ScoreRecord` 带 `operator`、`updatedAt`、`status`；写操作可选 `SetEvent` 发合约事件（后端订阅选做）。  
3. **链上只信多背书**：沿用 test-network 默认 **AND(Org1, Org2)** 于写操作；查询可走单 Peer。  
4. **应用层企业惯例**：Node 后端 **单例 Gateway**、`.env` 管理连接与钱包路径、**结构化日志**（请求 id + txId）、**REST 错误体**统一 `{ code, message, detail? }`。  
5. **验收物**：`scripts/demo.sh`（curl 顺序：录入 → 查询 → 更正 → 历史 ≥2 条 → 验真）、`实验报告.md` 中贴 **txId** 与截图。

### 13.3 链码演进顺序（仍在 `score-chaincode`，用 `deployCC -ccv 1.1 -ccs 2` 升级）

| 顺序 | 能力 | 说明 |
|------|------|------|
| 1 | **数据模型对齐 §3.1** | 在现有 JSON 上增加 `status`、`updatedAt`、`operator`（字符串即可）、可选 `remark` |
| 2 | **`CorrectScore` / `RevokeScore`** | 旧记录改状态 + 新记录或仅作废；**禁止物理删除** |
| 3 | **`GetScoreHistory`** | 封装 `GetHistoryForKey`，返回可 JSON 化的版本列表（txId、时间、值摘要） |
| 4 | **`VerifyScore`** | 入参「声称分数」与链上当前 `ACTIVE` 记录比对；可选增加 `VerifyScoreHash` |
| 5 | （可选）**`cid` 读 MSP/OU** | 仅区分 Org1/Org2 写权限，避免重 ABAC；答辩一句话即可 |

每步完成后在 VM 上：**`go mod tidy` → `vendor` → `deployCC` 升版**，并用 **`peer` + Args** 或后端脚本留证。

### 13.4 应用层顺序（`~/work/score-app` 或仓库 `backend/`）

| 顺序 | 产出 |
|------|------|
| B1 | **Express/Fastify** + `GET /api/health`、`GET /api/chain/summary`（summary 可先返回固定通道名 + 调 `network` 脚本或占位，后续接 `fabric-gateway`） |
| B2 | 接入 **Fabric Gateway**：`POST/GET` 成绩与 **txId** 回传前端 |
| B3 | **CSV 批量**：仅后端读文件，循环提交交易（失败行返回列表） |
| B4 | 前端 **Vue/React** 按 §12 路由做「角色切换 + 列表 + 详情 + 历史 + 验真」——**基线已落地**（仓库 **`frontend/`**，见 **§十二、§12.7**）；**详情动态路由**、**批量导入**、**完整 `chain/summary`** 仍为可选 |

开发机内存紧张时：**VM 只跑 Docker Fabric**；Node 可在 Windows 跑但演示前须在 VM 或 NAT 下验证浏览器能访问。

### 13.5 你现在「具体该怎么干」（按周也可压缩）

1. **冻结拓扑**：继续 **test-network + mychannel + LevelDB**，不扩 Peer、不加 CouchDB（除非老师要求富查询）。  
2. **改链码（优先）**：实现 §13.3 的 1→3（模型 + 更正/作废 + 历史）；CLI 或临时 `node` 脚本调通后再写完整后端。

5. **起前端**：仓库 **`frontend/`** 已覆盖 **Dashboard、查询、历史、录入、更正、作废、验真** 与角色菜单；答辩前配置 **`VITE_API_BASE`** 并补截图（见 **`实验报告.md` §6.4、§7.2**）。可选迭代：**`/scores/import`**、**`/api/chain/summary`**。

### 13.6 答辩一句话（建议背下来）

「我们在 **双组织单通道** 上模拟 **教务处与学院** 共同背书写入；链上成绩有 **状态与历史版本**，更正不做物理删除；应用层负责 **身份与界面**，链上负责 **存证与可审计**；监控与多 Orderer 属于生产扩展，本实验聚焦 **共识与背书下的业务正确性**。」

## 十四、宿主机与虚拟机多线推进（P3 起）

从 **P3（后端 + 前端）** 开始，建议**两条线并行**：**Windows 宿主机**负责编辑与界面迭代；**Ubuntu 虚拟机**负责 Fabric、链码容器与「与真实账本」的最终联调。这样既不违背「网络在 Linux VM 上跑」的验收习惯，又能减少在 VM 里开 IDE 的卡顿。

### 14.1 分工一览

| 工作项 | 建议位置 | 说明 |
|--------|----------|------|
| 代码编辑（链码、**`backend/`**、**`frontend/`**、文档） | **宿主机** | Cursor / VS Code；路径与换行按仓库规范即可。 |
| **Docker、test-network、Peer/Orderer、链码部署与 `peer` CLI** | **虚拟机** | 与课程要求、答辩演示一致。 |
| **后端进程（Node + Fabric Gateway）** | **优先虚拟机**（与链同机） | **`FABRIC_CRYPTO_BASE`** 指向 VM 内 **`…/org1.example.com`** 绝对路径，避免 Windows 路径与证书挂载错位。 |
| **前端静态页 / SPA 开发** | **宿主机** | 开发时将 API 基地址设为 **`http://<虚拟机 NAT IP>:<后端端口>`**（默认后端 **`3000`** 时即 **`http://<VM_IP>:3000`**）；浏览器在宿主机访问即可（NAT 端口转发由 VMware/VirtualBox 配置）。 |
| **答辩前整体验收** | **虚拟机内浏览器或宿主机指 VM IP** | 以「能连上 VM 上起的后端 + 账本」为准留截图。 |

### 14.2 同步方式（避免「拖文件夹丢权限」）

1. **首选 Git**：宿主机 `commit` / `push`，虚拟机 `clone` 或 `pull` 到 **`~/work/final-project`**（或课程统一目录），再按 **`score-chaincode/README-VM.md`** 做 **`chmod`** / **`vendor`**。  
2. **共享文件夹拖拽**：每次拖入后执行一次 **`chmod -R u+w`**（或文档中的 **`fixwork`**），否则 **`go mod vendor`** 可能报权限错误。  
3. **链码与网络**：改链码后仍在 **VM** 上 **`deployCC` 升版**；宿主机只改源码，不在 Windows 上跑完整 Fabric 网络（除非老师明确要求）。

### 14.3 联调顺序（建议）

1. VM：**`./network.sh up`** → **`createChannel`** → **`deployCC` score**（当前定义见实验报告 **§6.5 / §7.1**）。  
2. VM：**`cd backend`**，复制 **`.env.example`** 为 **`.env`**，填写 **`FABRIC_CRYPTO_BASE`**（Org1 `peerOrganizations` 根目录的**绝对路径**）。  
3. VM：**`npm install`** → **`npm run build`** → **`npm start`**；宿主机 **`curl http://<VM_IP>:3000/api/health`**。  
4. 若 **`submitTransaction`** 报背书相关错误：对照 **`peer chaincode invoke`** 的双 Peer **TLS** 参数，检查 Gateway 是否满足 **test-network** 默认背书策略（必要时在 **`backend/README.md`** 中记录调整）。  
5. 前端：仓库 **`frontend/`** 已按 **§12 / §12.7** 实现主流程；**生产演示**前在 **`frontend/.env.development`**（或构建用 **`.env.production`**）中填写真实 **`VITE_API_BASE=http://<VM_IP>:3000`**，用宿主机或 VM 内浏览器跑通一轮并截图写入 **实验报告 §6.4 / §7.2**。

### 14.4 可选：宿主机无账本时的 Mock

若宿主机暂时连不上 VM，可在 **`backend/`** 增加环境变量（如 **`USE_MOCK_FABRIC=true`**）走内存假数据，仅用于 UI 与接口契约开发；**链上行为与 T1～T5 证据仍以 VM + CLI 或真实 Gateway 为准**，报告里需区分「Mock 开发」与「真实联调」截图。
